**Goal:** Expo app reaches feature parity with web for the 3 main flows.

### A1 — Marketplace (mobile)
- [ ] Marketplace grid with cursor pagination (reuse `listListings` API client)
- [ ] Listing detail screen with photo gallery + WhatsApp contact
- [ ] Post listing form (photo upload, category picker, price)
- [ ] Sort + filter chips (reuse `FilterChips` logic)

### A2 — Chat (mobile)
- [ ] ConversationList screen (unread badge via SignalR)
- [ ] ChatRoom screen (MessageBubble, MessageComposer)
- [ ] SignalR hub connection in Expo (reuse shared `getHubConnection`)

### A3 — Groups (mobile)
- [ ] Groups list + group detail screen
- [ ] Event list within group
- [ ] Join/leave group action

### A4 — Push Notifications
- [ ] Expo `expo-notifications` integration
- [ ] Backend: `PushToken` table + `NotificationService` → Expo push API

**Success criteria:** A user on mobile can browse marketplace, chat, join groups, and receive push notifications.

## Wave C — PWA + SEO

**Goal:** Improve retention with offline support and improve discoverability.

### C1 — PWA
- [ ] `public/manifest.json` (name, icons, theme_color=#2563EB, display=standalone)
- [ ] Service worker via `next-pwa` or manual `public/sw.js`
- [ ] Offline fallback page (`/offline`)
- [ ] Cache strategy: stale-while-revalidate for API responses (5-min TTL)

### C2 — Performance (Lighthouse ≥ 90)
- [ ] Image optimization: convert remaining `<img>` to use explicit dimensions + `loading=lazy`
- [ ] Font subsetting: Outfit Latin only
- [ ] Bundle analysis: identify and code-split heavy imports
- [ ] `fetchpriority="high"` on above-fold images

### C3 — SEO workaround (static export limitation)
- [ ] Apache mod_rewrite: add OG meta fallback via `.htaccess` for crawlers
- [ ] OR: generate static OG pages at build time for listing + post preview routes
- [ ] `public/sitemap.xml` generated at build (listing IDs from API snapshot)
- [ ] `robots.txt` with correct Disallow for admin/onboarding routes

**Success criteria:** Lighthouse PWA score ≥ 90; app installable on Android; sitemap submitted to Google Search Console.

## Wave D — Monetização seed

**Goal:** Establish the first paid feature to validate willingness-to-pay.

### D1 — Business account type
- [ ] `AccountType` enum: `Resident | Business` on `User`
- [ ] Business profile fields: `BusinessName`, `BusinessCategory`, `BusinessPhone`, `BusinessWebsite`
- [ ] Business badge in feed + marketplace (distinct from Verified badge)
- [ ] Admin: promote user to Business (or self-serve with review)

### D2 — Business Spotlight
- [ ] `Spotlights` table: `BusinessId`, `BairroId`, `StartDate`, `EndDate`, `SlotIndex` (3 max per bairro)
- [ ] Feed API: inject up to 3 spotlight cards every N posts
- [ ] Spotlight card design: amber-tint bg, "Negócio local" label, CTA button
- [ ] Admin panel: assign/remove spotlights manually (Phase 1 — no payments yet)

### D3 — Analytics for businesses (basic)
- [ ] `ListingView` event logging (impression + click)
- [ ] Business dashboard: views / contacts / listings in 30 days
- [ ] Export: CSV download of analytics

**Success criteria:** 1 business account created manually; spotlight appears in feed; business can see their view count.
